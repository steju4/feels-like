  • 1. Übersicht der Softwarearchitektur

    1.1 Architekturmuster

    Basierend auf den im Software-Entwicklungsplan (SDP) festgelegten Technologien (React für das Frontend, Node.js/Express für das Backend, SQL für die Datenbank) wählen wir eine klassische 3-Schichten-Architektur (3-Tier-Architecture).

    Diese Architektur trennt klar zwischen Präsentation, Geschäftslogik und Datenhaltung, was die Wartbarkeit erhöht und paralleles Arbeiten im Team (Frontend/Backend) ermöglicht.

    • Präsentationsschicht (Oberflächenpaket): React (Single Page Application). Verantwortlich für die Darstellung (UI) und Benutzerinteraktion.
    • Logikschicht (Fachklassenpaket): Node.js mit Express. Verantwortlich für die API-Bereitstellung, Verarbeitung der Geschäftslogik und Authentifizierung.
    • Datenhaltungsschicht: SQL-Datenbank (SQLite). Verantwortlich für die persistente Speicherung der Entitäten.

    Beschreibung der Pakete:

    1. Oberflächenpaket (Implementiert im Client)
      • Views (Komponenten): Hier befinden sich die Benutzeroberflächen wie LoginView (Authentifizierung) und TrainingView (Datenerfassung).
      • API Client: Eine Hilfsschicht (api.js), die die Kommunikation zum Backend kapselt und REST-Anfragen mittels axios durchführt.
    1. Fachklassenpaket (Implementiert im Server/Services)
      • Business Services: Hier liegt die eigentliche Logik (TrainingService, AnalyseService). Sie sind entkoppelt von der HTTP-Schnittstelle.
      • Controller: Die technischen Einstiegspunkte (z. B. authController.js). Sie nehmen HTTP-Anfragen entgegen, validieren diese und rufen die Services auf.
    1. Datenhaltung (Implementiert im Server/Models)
      • Data Access / Models: Klassen wie AthletRepository (Sequelize Models), die das Laden und Speichern aus der Datenbank übernehmen.

    2.4 Middleware-Module

    authMiddleware

    • Datei: server/src/middleware/authMiddleware.js
    • Aufgabe: Schützt Routen vor unautorisiertem Zugriff. Wird als Express-Middleware vor geschützten Route-Handlern eingehängt.
    • Ablauf:
      1. Liest Token aus Authorization: Bearer <token> (bevorzugt) oder alternativ aus HttpOnly-Cookie token.
      1. Verifiziert den Token mit JWT_SECRET.
      1. Bei Erfolg: req.user = { id, role, name }.
      1. Bei Fehler: Cookie wird geloescht und HTTP 401 zurueckgegeben.
    • Eingabe: HTTP-Request mit Authorization Header (Bearer <token>) oder HttpOnly-Cookie (token)
    • Ausgabe: req.user = { id, role, name } . Bei ungültigem Token: Cookie wird gelöscht und HTTP 401 zurückgegeben.

  • 3. Technische Spezifikationen

    3.1 Framework-Entscheidungen

    • Kommunikation: REST API (JSON Format).
    • Authentifizierung: Erfolgt über JWT. Primär wird ein HttpOnly-Cookie (token) für Browser-Sessions verwendet. Zusätzlich wird für API-Clients ein Authorization Header im Format Bearer <token> unterstützt.
    • Fehlerbehandlung: Exceptions werden in der Controller-Schicht abgefangen und in standardisierte HTTP-Statuscodes (500, 400, 404) umgewandelt.

    3.2 Verwendete Bibliotheken & Tools

    Um die Architektur umzusetzen, werden folgende spezifische Bibliotheken genutzt:

    Frontend (Client):

    • vite: Build-Tool und Entwicklungsserver.
    • react-router-dom: Für das client-seitige Routing (SPA-Navigation).
    • axios: HTTP-Client für die Kommunikation mit der REST-API.

    Backend (Server):

    • express: Webserver-Framework.
    • sequelize: ORM zur Abstraktion der SQL-Befehle.
    • sqlite3: Datenbank-Treiber für SQLite.
    • bcrypt: Sicheres Hashing von Passwörtern.
    • dotenv: Verwaltung von Umgebungsvariablen und Secrets.
    • nodemailer: Nodemailer wird für den Versand von Einladungs-E-Mails eingesetzt. Der Versand ist über konfigurierbare Modi umgesetzt (auto, smtp, ethereal, console). Im lokalen Entwicklungsbetrieb wird standardmäßig Ethereal genutzt, um den E-Mail-Flow zu testen, ohne SMTP-Zugangsdaten vorauszusetzen. Für produktionsnahe Szenarien kann ohne Codeänderung auf SMTP umgestellt werden.
  • 4. Qualitäts- und Sicherheitsaspekte

    4.1 Sicherheitsmaßnahmen

    • Passwort-Sicherheit: Passwörter werden niemals im Klartext gespeichert. Beim Anlegen oder Ändern eines Users wird das Passwort automatisch über Sequelize Lifecycle Hooks (beforeCreate, beforeUpdate) mittels bcrypt gehasht und gesalzen.
    • Authentifizierung: Wir verwenden HTTP-Only Cookies für die Übertragung von JWTs (JSON Web Tokens). Dies verhindert, dass client-seitiges JavaScript (z.B. bei XSS-Angriffen) auf den Token zugreifen kann.
    • Secrets Management: Sensitive Daten wie JWT_SECRET und Datenbank-Zugangsdaten werden nicht im Code hardcodiert. Sie werden über .env Dateien geladen, die per .gitignore strikt von der Versionsverwaltung ausgeschlossen sind.
    • Route Protection: Im Backend prüft authMiddleware bei geschützten Requests die Gültigkeit des Tokens. Dabei werden sowohl Authorization Header (Bearer <token>) als auch das HttpOnly-Cookie (token) unterstützt. Im Frontend verhindert ProtectedRoute, dass nicht eingeloggte Nutzer auf interne Seiten gelangen.
    • E-Mail Versand: In der Entwicklungs- und Testumgebung wird Ethereal als Test-Mail-Provider eingesetzt, um unbeabsichtigten Versand an reale Empfänger zu vermeiden und dennoch einen prüfbaren Nachweis des Einladungsversands zu ermöglichen (Preview-URL)
    • Passwort-Reset: Reset-Token mit Typkennzeichnung und begrenzter Laufzeit (konfigurierbar, Standard 30 Minuten). Schutz vor Token-Wiederverwendung durch Bindung an einen Fingerprint des aktuellen Passwort-Hashs. Generische Antwort bei Reset-Anforderung zur Reduzierung von Account-Enumeration im UI-Flow

    4.2 Entwicklungsprozess

    • Version Control: Nutzung von Git mit Feature-Branches.
    • Separation of Concerns: Strikte Trennung von Routing, Controller und Service-Logik zur Gewährleistung der Testbarkeit und Wartbarkeit.

    4.3 Entwurfsmuster (Design Patterns)

    Zur Sicherstellung einer sauberen Architektur und Wartbarkeit werden im Projekt zwei zentrale Design Patterns eingesetzt:

    1. Singleton Pattern (Bereich: Infrastruktur)

    • Kontext: Die Anwendung benötigt Zugriff auf geteilte Ressourcen, die nur einmal existieren dürfen (Datenbankverbindung, Logging-Instanz).
    • Problem: Das mehrfache Öffnen von Datenbankverbindungen bei jedem Request würde zu Performance-Problemen und Inkonsistenzen führen.
    • Lösung: Wir implementieren die Datenbank-Instanz (sequelize) und den API-Client (axios im Frontend) als Singletons. In Node.js nutzen wir dazu das Modul-System (Caching), das sicherstellt, dass beim Importieren der Datei db.js immer dieselbe Instanz zurückgegeben wird.

    2. Strategy Pattern (Bereich: Geschäftslogik / Ranking)

    • Kontext: Die Anwendung muss verschiedene Rankings berechnen (z. B. "Meiste Kilometer", "Meiste Trainingseinheiten", "Längste Dauer").
    • Problem: Ein Controller, der all diese Varianten mit if (type == 'km') ... else if (type == 'time') ... abhandelt, wird schnell unübersichtlich und schwer zu testen (Verstoß gegen das Open/Closed Principle).
    • Lösung: Wir kapseln die Berechnungslogik in austauschbare Strategien.
      • Der RankingController definiert nur den Input (die Trainingsdaten).
      • Es gibt separate Strategie-Funktionen (z. B. DistanceStrategy, FrequencyStrategy).
      • Zur Laufzeit wird basierend auf dem User-Filter die passende Strategie gewählt, ohne den Controller-Code ändern zu müssen.
