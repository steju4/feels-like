1. Einleitung & Zielsetzung

Dieses Dokument beschreibt die Strategie zur Qualitätssicherung und Verifikation der Software "Feels Like".
Ziel ist es, sicherzustellen, dass die implementierten Funktionen den Anforderungen (aus der Anforderungsanalyse) entsprechen und das System stabil läuft.

2. Anwendungsbereich (Scope)

Verifiziert werden folgende Komponenten des Systems (gemäß SDD):

  1. Backend (Server):
    • REST-API Endpunkte (Authentifizierung, Athletenverwaltung, Training).
    • Datenbank-Integrität (SQLite/Sequelize Models).
    • Geschäftslogik (z. B. Ranking-Berechnungen).
  1. Frontend (Client):
    • Benutzeroberfläche (Login, Dashboard, Formulare).
    • Client-seitige Validierung.
    • Routing und Navigation.

3. Verifikationsstrategie

Wir wenden eine Kombination aus statischen und dynamischen Testverfahren an, wie in der Vorlesungseinheit 2.4 beschrieben.

3.1 Konstruktive Qualitätssicherung (Statisch)

  • Code-Richtlinien: Nutzung von ESLint (bereits konfiguriert), um Syntaxfehler und Stilbrüche automatisch zu erkennen.
  • Peer Reviews: Es gilt das Vier-Augen-Prinzip. Kein Feature-Branch wird in den dev Branch gemerged, ohne dass ein anderes Teammitglied den Code auf GitHub gesichtet und freigegeben hat (Pull Request Review).
  • Projektmanagement (GitHub Project Board): Die Entwicklung wird agil in Sprints organisiert. Aufgaben werden als GitHub Issues im GitHub Project Board des Repositories erfasst und den Spalten Backlog → Ready → In Progress → In Review → Done zugeordnet. Jedes Issue ist einem Sprint zugewiesen und wird mit einem Feature-Branch verknüpft (z. B. feature/training-crud). Kein Feature-Branch wird direkt in main gemerged — der Übergang von In Review nach Done erfolgt erst nach bestandenem Pull Request Review..

3.2 Testmethoden (Dynamisch)

A. Unit Tests

  • Testen einzelner Funktionen in Isolation.
  • Fokus: Komplexe Logik im Backend (z. B. RankingService, siehe SDD).
  • Methode: Äquivalenzklassentests (siehe 3.3).

B. Integrationstests / API-Tests

  • Testen der Schnittstellen zwischen Frontend und Backend.
  • Tool: Postman oder Thunder Client.
  • Szenario: Login durchführen -> Token erhalten -> Mit Token geschützte Route aufrufen.

C. Systemtests (Manuell)

  • Black-Box-Tests der gesamten Anwendung über die Benutzeroberfläche.
  • Prüfung gegen die Use Cases (z. B. "Als Athlet möchte ich ein Training erfassen").

4. Verifikationsumgebung

4.1 Hardware

  • Entwickler-Laptops (Windows/Mac/Linux) als lokale Testumgebung.

4.2 Software & Tools

  • IDE: Visual Studio Code.
  • Laufzeitumgebung: Node.js (v18+).
  • Datenbank: SQLite (lokal).
  • API-Testing: Postman / cURL (voraussichtlich!)
  • Linter: ESLint (statische Analyse).
  • Browser: Google Chrome / Firefox (für Frontend-Tests).

5.2 Testausschlüsse

Noch nicht durch Tests abgedeckt (Stand: 05.03.2026):

  • authController (Login/Logout) -- kein Unit-Test vorhanden
  • userController (profilLaden) -- kein Unit-Test vorhanden
  • authMiddleware (JWT-Verifizierung) -- kein Unit-Test vorhanden
  • Frontend-Komponenten -- keine automatisierten Tests
  • Integrationstests (API-Endpunkte mit Datenbank)
  • E2E-Tests (vollständiger User-Flow)

6. Rollen & Verantwortlichkeiten

Damit die Qualitätssicherung nicht vergessen wird, werden folgende Rollen zugewiesen:

  • Entwickler (Author): Erstellt den Code und führt lokale Tests durch (Schreibt ggf. Unit-Tests).
  • Reviewer: Prüft den Pull Request eines anderen Entwicklers auf Logikfehler und Einhaltung der Coding Standards.
  • Tester: Führt manuelle Systemtests vor einem Release durch.

7. Zeitplan und Meilensteine

Die Verifikation erfolgt begleitend zur Entwicklung ("Continuous Verification"). Da das Projekt agil (in Sprints) durchgeführt wird, passen sich die konkreten Testinhalte den jeweiligen Entwicklungszielen an.

7.1 Phasenmodell der Verifikation

Der Verifikationsprozess folgt dem Fortschritt der Implementierung:

PhaseZeitraumFokus der VerifikationMethoden
Phase 1: InitialisierungProjektstart (Sprint 0/1)Sicherstellung der Entwicklungsfähigkeit. Setup der Linter (ESLint), der Test-Datenbank und der CI/CD-Grundlagen.Konfiguration, erste Tests
Phase 2: Komponenten-TestsFortlaufendPrüfung neuer Module direkt nach der Erstellung. Ziel ist es, Logikfehler in Services oder Helper-Funktionen früh zu finden.Unit Tests, Code Reviews
Phase 3: Integrations-TestsNach Feature-ImplementierungPrüfung des Zusammenspiels von Client (Frontend) und Server (API). Werden Daten korrekt übertragen und gespeichert?API-Tests (Postman), Manuelle Tests
Phase 4: System-TestsVor Meilenstein-AbgabenEnd-to-End Prüfung der gesamten Anwendung aus Nutzersicht. Funktionieren die Abläufe (Workflows) durchgängig?Manuelle Tests
Phase 5: AbschlussProjektendeFinaler Review aller Artefakte und Prüfung der Traceability (Wurden alle Anforderungen getestet?).Audit, Abschlussbericht

7.2 Verifikations-Meilensteine

Die Meilensteine definieren den qualitativen Zustand der Software am Ende eines Zeitabschnitts:

Sprint 1: Infrastruktur & Basis-Funktionalität

  • Ziel: Das System ist grundlegend lauffähig ("Walking Skeleton").
  • Verifikations-Fokus:
    • Stabilität der Architektur (Client-Server-Verbindung).
    • Sicherheit der Authentifizierung (Login/Registrierung, Token-Handling).
    • Korrekte Anbindung der Datenbank.
  • Methode: Code Reviews und exploratives Testen der Basis-Routen.

Sprint 2-3: Erweiterung der Kernfunktionen

  • Ziel: Die Hauptanwendungsfälle (Use Cases) sind funktional implementiert.
  • Verifikations-Fokus:
    • Datenintegrität bei CRUD-Operationen (Erstellen, Lesen, Aktualisieren, Löschen von Datensätzen).
    • Validierung von Benutzereingaben (Fehlerbehandlung bei ungültigen Daten).
  • Methode: Durchführung definierter Integrations-Tests für alle neu hinzugefügten API-Endpunkte.

Sprint 4: Finalisierung & Logik

  • Ziel: Das System ist vollständig, stabil und bereit für die Abnahme.
  • Verifikations-Fokus:
    • Prüfung komplexer Geschäftslogiken und Berechnungen.
    • Usability-Tests der Benutzeroberfläche (UX).
    • Grenzfall-Analysen (Edge Cases).
  • Methode: Vollständiger Systemtest aller Use Cases und Regressionstests (Sicherstellen, dass alte Funktionen noch gehen).

8. Dokumentation & Nachverfolgbarkeit

Um sicherzustellen, dass keine Anforderung vergessen wurde, wird eine einfache Traceability-Matrix gepflegt.

  • Verfahren: Jeder definierte Testfall (TC-ID) wird einer Anforderung (UC-ID aus dem Pflichtenheft) zugeordnet.
  • Fehler-Tracking: Gefundene Bugs werden als GitHub Issues erfasst und mit dem Label bug versehen. Im Pull Request wird referenziert, welcher Testfall fehlgeschlagen ist.
